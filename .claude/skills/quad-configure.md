# QUAD Configuration Wizard

This skill interactively configures the QUAD platform by asking questions about the user's environment, then writes two files:
- `quad.toml` — all SDK paths, platform settings, and runtime configuration
- `.env` — all secrets, API keys, and sensitive values (never committed)

## Step 1: Gather Context

Before asking questions, read the current state:
1. Read `quad.toml` (if it exists) to know current values
2. Read `configs/quad.toml.example` for the full config schema
3. Read `.env` (if it exists) to know what keys are already set
4. Read `.env.example` to know what keys are expected
5. Check `src/quad/adapters/dsp_env.py` for chipset → Hexagon version lookup

## Step 2: Ask Configuration Questions

Use `AskUserQuestion` to ask the following groups of questions. Only ask what is needed — if a value is already set and seems correct, confirm rather than re-ask.

### Group 1: Adapter Mode
Ask:
- **Adapter mode**: mock (development, no hardware needed) or real (use actual Qualcomm SDKs)

### Group 2: QAIRT/SNPE SDK (only if real mode selected)
Ask:
- **QAIRT SDK path**: Where is the SDK installed? (e.g. `/path/to/qairt/2.45.0.260326`)
- **SDK version**: Confirm detected or ask for version string

### Group 3: Target Device
Ask (multi-select what platforms they're using):
- **Platforms**: Windows (Snapdragon X Elite), Linux (QCS2210/Arduino UNO Q), Android, None/Mock only
- For Linux: SSH host IP, SSH username
- For Android: ADB device serial (leave blank for auto-detect)
- **Hexagon/DSP version**: Which Hexagon DSP version? (auto-detect from chipset or ask)

### Group 4: API Keys & Secrets
Ask:
- **Qualcomm AI Hub API key**: (will go to .env, never quad.toml)
- **Optional**: Any other service keys

### Group 5: Performance Preferences
Ask:
- **Log level**: debug / info / warning
- **Default power mode**: performance / balanced / efficiency

## Step 3: Write Files

### Write `quad.toml`

Generate the complete `quad.toml` from answers. Use this structure:

```toml
# QUAD Configuration
# Auto-generated by quad-configure skill
# Last updated: {date}

[server]
adapter_mode = "{mock|real}"
log_level = "{level}"
log_format = "console"
model_output_dir = "./output"
template_dir = "./templates"
default_power_mode = "{mode}"

[adapters.qairt]
sdk_path = "{path}"
version = "{version}"

[adapters.qnn]
sdk_path = "{path}"    # Usually same as qairt

[adapters.snpe]
sdk_path = "{path}"    # Usually same as qairt

[adapters.hexagon]
sdk_path = ""
hexagon_version = "{v65|v66|v68|v69|v73|v75}"

[adapters.ai_hub]
api_key_env = "QAI_HUB_API_KEY"   # Key lives in .env

[platforms.windows]
enabled = {true|false}
device_type = "local"

[platforms.linux]
enabled = {true|false}
device_type = "remote"
ssh_host = "{ip}"
ssh_user = "{user}"
ssh_key = "~/.ssh/id_rsa"

[platforms.android]
enabled = {true|false}
device_serial = "{serial}"
adb_path = "adb"

[dsp]
hexagon_version = "{v65|v66|v68|v69|v73|v75}"
adsp_lib_dir = ""   # Auto-set during deployment
target_os = "android"  # android | linux | automotive
```

### Write `.env`

Generate `.env` with all secrets. NEVER write secrets to `quad.toml`.

```bash
# QUAD Environment Variables
# Auto-generated by quad-configure skill — DO NOT COMMIT
# Copy template: cp .env.example .env

# Qualcomm AI Hub API key
QAI_HUB_API_KEY={key or blank}

# SSH credentials (if using remote Linux target)
TARGET_IP={ip or blank}
TARGET_USER={user or blank}
TARGET_SSH_KEY={path or blank}

# Android (ADB)
ANDROID_SERIAL={serial or blank}

# QAIRT SDK (can also be set via quad.toml)
QAIRT_SDK_ROOT={path or blank}
QNN_SDK_ROOT={path or blank}
SNPE_ROOT={path or blank}

# Hexagon SDK
HEXAGON_SDK_ROOT={path or blank}

# Android NDK (for UDO compilation)
ANDROID_NDK_ROOT={path or blank}
```

### Update `.env.example`

Always update `.env.example` to match `.env` structure but with blank/placeholder values. This file IS committed to git.

### Update `.claude/settings.json`

If real mode was selected, update the `QUAD_ADAPTER_MODE` env in the MCP server config.

## Step 4: Validate

After writing files:
1. Run `python -m quad.cli.main doctor` to validate the configuration
2. Report any issues found
3. Suggest next step: `./launch.sh --real` or `./launch.sh --mock`

## Step 5: Summary

Print a summary table showing what was configured, what's still missing, and the exact commands to get started.
